# KeepiQ Codebase Guide

## 1. What this project is

KeepiQ is an MVP for personal incoming-information management inside Telegram.

The product promise is simple:

- user sends anything into the bot
- the system keeps it safe immediately
- the system tries to understand whether it is one task, a list, or something unclear
- the bot answers in natural Russian
- the user gets exactly 4 concrete action buttons
- if the system is not confident, the message still stays searchable in Inbox

For MVP there are only three user-visible buckets:

- task / reminder
- list
- inbox

There are no separate entities for notes, ideas, events, tags, or projects.

## 2. Decisions and assumptions fixed for this MVP

These are places where the product brief left room for interpretation and the code takes a practical choice.

### Product assumptions

- Default timezone is `Europe/Moscow`.
- If a date is found without time, default time is `10:00` local time.
- If the message looks like an event, ticket, booking, or reservation, the main reminder is scheduled `3 hours before`.
- Otherwise, if a task has a concrete time and it is far enough in the future, the reminder is usually scheduled `1 hour before`.
- If AI is unavailable, the item is still stored and marked for retry.
- If OCR or transcription is unavailable, the app stores whatever text it still has: caption, raw text, file context.
- Search works without vector storage. Embeddings are optional and used only as reranking help.
- Mini App local development can use `/auth/dev` when `ALLOW_DEV_AUTH=true`.
- Frontend uses `HashRouter` to avoid SPA rewrite issues on static hosting.

### Architecture assumptions

- Background jobs are implemented with a separate APScheduler-based worker process instead of Redis/Celery to keep MVP deployment simple.
- Bot can run in webhook mode or local polling mode.
- The API is the source of truth for Mini App data.
- Bot button actions are persisted in `bot_actions`, not encoded fully inside `callback_data`, because Telegram callback payloads are too short for rich actions.

## 3. End-to-end data flow

### Incoming Telegram message -> saved object

1. `backend/app/bot/handlers.py`
   - catches any incoming Telegram message
   - identifies Telegram user
   - calls the extractor for text / voice / photo / document

2. `backend/app/bot/extractor.py`
   - maps Telegram message kinds into `source_kind`
   - downloads temporary media when needed
   - passes audio and images to `MediaExtractionService`

3. `backend/app/services/inbox.py`
   - creates `InboxItem` immediately
   - saves raw text, extracted text, Telegram chat/message ids

4. `backend/app/services/ai_pipeline.py`
   - runs structured understanding through `AIClient`
   - runs backend date parsing through `DateParsingService`
   - merges both, with backend fallback rules

5. `backend/app/services/actions.py`
   - persists exactly 4 action records in `bot_actions`
   - one of them is always `keep_in_inbox`

6. `backend/app/copy/templates.py`
   - builds the final human bot text
   - final wording never comes from raw LLM prose

7. `backend/app/bot/keyboards.py`
   - renders the 4 inline buttons

8. User taps a button
   - callback resolves `BotAction`
   - service converts the inbox item into a task or list, or keeps it in Inbox
   - the service message with buttons is deleted
   - bot sends a short natural confirmation

## 4. How the AI pipeline works

The AI flow is intentionally two-stage.

### Stage A. Preprocessing

Handled by:

- `backend/app/bot/extractor.py`
- `backend/app/services/media_extraction.py`

Responsibilities:

- text: keep as-is
- voice: transcribe
- image/screenshot: OCR/vision extraction
- forwarded messages: keep forwarded text context
- links: keep link text; future version can enrich title/summary with fetchers

Binary files are not stored in the database. They are downloaded temporarily, processed, and deleted.

### Stage B. Structured understanding

Handled by:

- `backend/app/ai/prompts.py`
- `backend/app/ai/schemas.py`
- `backend/app/ai/client.py`
- `backend/app/services/ai_pipeline.py`

Rules:

- LLM returns JSON only
- JSON is validated with Pydantic
- backend merges AI with deterministic date parsing
- if JSON is bad or incomplete, fallback heuristics are used
- final user-facing text is generated by backend templates

### Why this is important

This prevents bad bot language and makes the system stable even if the model is slightly inconsistent.

## 5. Date parsing and reminder rules

Handled by:

- `backend/app/services/date_parsing.py`
- `backend/app/core/clock.py`
- `backend/app/services/action_templates.py`
- `backend/app/services/tasks.py`

Important behavior:

- Russian natural phrases like `завтра`, `вечером`, `после обеда`, weekdays, and explicit times go through `dateparser`
- backend normalizes soft hints like `вечером -> 19:00`
- if only a date is found, due time becomes `10:00`
- action buttons can also create date values, for example `Напомнить завтра в 10:00`
- reminders are single-shot only in MVP

## 6. Search design

Handled by:

- `backend/app/search/service.py`
- optional embeddings from `backend/app/ai/client.py`

Current search strategy:

1. gather candidates from tasks, lists, inbox items
2. lexical ranking over title, summary, description, normalized text
3. optional embedding rerank if embeddings are enabled and stored
4. return a mixed ordered result set

This gives human-friendly search without requiring pgvector in the first version.

## 7. Bot behavior

Handled by:

- `backend/app/bot/handlers.py`
- `backend/app/bot/keyboards.py`
- `backend/app/copy/templates.py`

Rules:

- tone stays calm and short
- after each analysis there are exactly 4 buttons
- buttons are concrete actions, never vague prompts
- service button message is deleted after selection
- reminder messages include quick actions: done, 1 hour, today evening, tomorrow 10:00

## 8. Mini App behavior

Handled by the React app in `frontend/src/`.

Screens:

- Today
- Inbox
- Calendar
- Lists
- Search

Key UX choices:

- mobile-first layout
- strong bottom navigation
- Telegram theme params are respected when available
- editing is inline and simple for MVP
- Inbox is treated as a safety net, not a trash bucket

## 9. Background jobs

Handled by:

- `backend/app/workers/scheduler.py`
- `backend/app/workers/run.py`

Jobs:

- retry failed/pending AI processing every 3 minutes
- due reminders every 1 minute
- morning summaries every 15 minutes
- evening summaries every 15 minutes

Summary timing is evaluated in each user's local timezone.

## 11. Safe extension path for V2

If you later add notes, ideas, events, or projects, the safest path is:

1. keep Inbox unchanged
2. add new entity tables separately
3. extend `DetectedType` and the AI schema
4. add new action templates only after backend conversion logic exists
5. keep backend-generated user copy as the final output source

Do not overload `Task` with too many unrelated meanings. Add separate entities once a new object has materially different behavior.

## 15. Security model

- every domain object is linked to `user_id`
- Mini App auth validates Telegram `initData`
- API endpoints always resolve current user from JWT and query by `user_id`
- bot actions are also bound to `user_id`
- no route returns cross-user data

## 16. What to inspect first when changing behavior

- parsing feels wrong: `backend/app/services/date_parsing.py`
- bot answers sound wrong: `backend/app/copy/templates.py`
- buttons are wrong: `backend/app/services/action_templates.py`
- AI returns weird structure: `backend/app/ai/prompts.py` and `backend/app/ai/schemas.py`
- search misses results: `backend/app/search/service.py`
- reminders are off: `backend/app/services/tasks.py` and `backend/app/services/summaries.py`
- Mini App layout needs UI work: `frontend/src/components/` and `frontend/src/pages/`

## 17. Final note

The code is intentionally biased toward reliability over aggressive automation:

- safe Inbox first
- strict JSON from AI
- backend owns final copy
- deterministic fallbacks for dates and actions
- user never loses incoming information even when AI fails

That is the core design principle of the MVP.
